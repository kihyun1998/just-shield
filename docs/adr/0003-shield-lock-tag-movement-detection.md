# shield.lock: 태그→SHA 박제 락파일로 태그 이동(하이재킹)을 탐지

`just-shield lock`이 각 액션의 태그→SHA 대응을 저장소 내 `shield.lock` 파일로 기록하고, 이후 스캔은 현재 대응을 박제본과 대조한다. 기존 버전 태그는 정상 상황에서 움직이지 않으므로, 이동 자체가 TeamPCP식 태그 하이재킹의 강한 사실 신호다(ADR-0002의 사실 규칙에 부합). 권고 DB 등재 이전, 즉 제로데이 창에서 작동하는 유일한 정적 탐지 수단이며 zizmor 등 기존 도구에 없는 차별점이다.

설계 영감은 인증서 투명성(Certificate Transparency)의 "불변이어야 할 것의 변화를 감시"하는 모델. 상태를 도구 내부가 아닌 저장소의 락파일에 두므로(npm package-lock 패턴) 도구는 무상태로 유지되고, 락파일 변경이 PR 리뷰 대상이 된다 — 신뢰 변경이 코드 리뷰를 통과해야 하는 구조.

트레이드오프: 사용자가 락파일을 커밋하고 갱신 흐름(`lock` 재실행)을 받아들여야 하는 운영 부담이 생긴다. 락파일 없이도 R1~R10은 동작하므로 점진 도입이 가능하다.

## Consequences

- 의도적 버전 업데이트와 하이재킹을 구분하는 수단이 필요하다 — 락파일 갱신은 명시적 `lock` 재실행으로만 일어난다.
- v2의 런타임 층(ⓒ 유출 차단, ⓓ 카나리아)과 독립적이다.
