# 배포 전략: 검증 가능한 바이너리가 토대, Marketplace가 주 채널, VS Code는 후순위

배포는 4단계 순서로 간다 — ⓪ 릴리스 자동화(크로스 플랫폼 바이너리 + SHA256SUMS + GitHub 빌드 출처 증명) → ① crates.io(`cargo install`) → ② GitHub Action 래퍼 + Marketplace 등록(SARIF 동반, 주 채널) → ③ 편의 채널(Homebrew/Scoop/winget/ghcr.io) → ④ VS Code 확장(v2 시기).

근거: 우리 도구의 배포 자체가 공급망이므로, 도구가 설파하는 원칙(체크섬·출처 증명·SHA 핀)을 배포 경로가 먼저 지켜야 한다 — 검증 가능한 바이너리(⓪) 없이는 래퍼(②)도 편의 채널(③)도 신뢰의 근거가 없다(신뢰의 사슬). 주 채널이 Marketplace인 이유: 사용자가 정의상 GitHub Actions 사용자다. VS Code 확장을 뒤로 미룬 이유: 별도 언어·마켓 운영의 큰 공사인 데다, TeamPCP가 VS Code 확장을 오염시킨 전례가 있어 가장 신중해야 할 채널이다. 래퍼는 엔진/포장 분리(질문 4, ADR-0004)에 따라 ⓪의 바이너리를 체크섬 검증 후 받는 얇은 껍데기로 유지한다.

릴리스 자동화는 cargo-dist 같은 서드파티 빌드 도구 없이 직접 작성한 워크플로로 한다 — 공식 바이너리를 만드는 사슬에 서드파티를 넣지 않는다(의존 크레이트 0개와 같은 논리). 우리 빌드는 의존성 0의 단일 바이너리라 잃는 편의도 가장 작고, cargo-dist의 `curl | sh` 인스톨러는 R3가 경고하는 바로 그 패턴이라 어차피 쓰지 않는다.

## Consequences

- "왜 VS Code 확장이 없냐"는 요청에는 이 ADR로 답한다 — 없는 게 아니라 마지막 순서.
- 각 채널 추가 시 "사용 예시가 우리 규칙(SHA 핀, 다이제스트 핀)을 통과하는가"를 출고 조건으로 둔다.
