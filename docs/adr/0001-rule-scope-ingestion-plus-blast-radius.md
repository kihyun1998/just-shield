# 검사 규칙 범위: 섭취 검증 + 피해 반경, 일반 워크플로 린터로는 확장하지 않음

just-shield v1의 검사 규칙은 10개로 확정한다 — 섭취 검증 7개(가변 참조 R1, 타이포스쿼팅 R2, 미검증 외부 도구 설치 R3, 이미지 다이제스트 미고정 R4, 임포스터 커밋 R5[온라인 옵션], 알려진 감염 버전 대조 R9[동봉 권고 DB 스냅숏 + `--online` 시 최신화], 쿨다운 R10[온라인 옵션, 기본 7일]) + 피해 반경 축소 3개(서드파티 액션의 시크릿 노출 R6, permissions 과잉 R7, `pull_request_target`+외부 체크아웃 R8). 스크립트 인젝션, 캐시 포이즈닝, 봇 조건 스푸핑 등 일반 워크플로 보안 규칙은 **의도적 비범위(non-goal)** 다.

근거: TeamPCP 캠페인 분석(Trivy 태그 하이재킹 76/77, KICS 전 태그 오염, `aquasecurtiy[.]org` 타이포스쿼팅) 결과 Tier 1 규칙이 실제 공격 경로를 직접 차단하고, 피해 규모를 키운 것은 과잉 권한 CI 토큰이었으므로 Tier 2가 필요하다. 반면 일반 린터 영역은 zizmor(39개 규칙)가 이미 잘 커버하므로, 그쪽으로 확장하면 정체성을 잃고 열화 복제품이 된다. "왜 인젝션 검사가 없나"라는 미래의 질문에 대한 답: 없는 게 아니라 안 하기로 한 것 — zizmor를 병행 사용하라.

## 보증 한계

SHA 핀은 불변성을 보증할 뿐 무해성을 보증하지 않는다. 미공표 오염(제로데이)은 정적 섭취 검증으로 원리적으로 탐지 불가하며 v1 보증 범위 밖이다 — 진짜 차단은 런타임 탐지(v2 후보)의 몫이고, v1에서는 쿨다운 R10으로 위험 창(발행 직후 미검증 기간)을 회피하는 것까지만 한다. 쿨다운 기본 7일은 "오염은 보통 며칠 내 발각된다"는 관찰에 근거하며, 보안과 패치 신속성의 트레이드오프다. 신뢰 앵커는 GitHub 플랫폼 무결성이다.

## Consequences

- 새 규칙 제안이 오면 "받는 산출물의 진위 검증인가, 오염 시 피해 축소인가"를 먼저 묻는다. 둘 다 아니면 거절.
- 오프라인 동작이 기본값이며, 네트워크가 필요한 규칙(R5)은 명시적 옵트인 뒤에만 실행된다.
