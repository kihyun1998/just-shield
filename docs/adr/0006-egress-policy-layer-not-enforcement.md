# v2 런타임 층(ⓒ)은 집행이 아니라 정책을 만든다 — egress 정책층

ⓒ "못 나가게" 층을 **관찰 → 허용 목적지 박제(락파일) → 대조** 의 정책층으로 구현한다. 자체 차단 근육(eBPF/iptables 인-러너 에이전트)은 만들지 않는다.

근거: ① 러너 *안에서* 도는 차단기는 공격자가 러너 안에서 root를 얻으면 꺼버릴 수 있는 구조적 한계가 있다 — 경비원이 도둑과 같은 방에 있는 모델이며, 선행자(step-security harden-runner)도 공개 우회 사례를 겪었다. ② GitHub이 러너 VM 바깥 인프라 레벨의 네이티브 egress 방화벽을 2026 로드맵에 올렸다 — 집행은 플랫폼이 흡수하는 중이다(commoditize). ③ 인프라가 바뀌어도 살아남는 것은 "무엇을 허용할지"라는 정책이고, 이를 리뷰 가능한 파일에 박제하는 것은 shield.lock에서 검증한 우리의 정체성이다(ADR-0003). 집행자가 누구든(우리의 감사 모드, harden-runner, 네이티브 방화벽) 허용 목록의 진실은 저장소 안 파일에 있게 한다.

대안과 기각: (가) 자체 가드 구현 — 성숙한 경쟁자와 곧 나올 네이티브 기능 사이의 셋째 바퀴, 적대적 영역(우회 공방)을 1인 프로젝트가 감당. (다) ⓒ 보류 — 네이티브 방화벽도 결국 "허용 목록"을 입력으로 요구하므로 정책 생산 문제는 사라지지 않는다.

## Consequences

- 관찰 모드는 회피 가능하다 — 보안 경계(security boundary)가 아니라 정책 작성을 위한 가시성 도구임을 문서에 명시한다. 강제 차단이 필요한 사용자에게는 harden-runner·네이티브 방화벽 병용을 안내한다.
- 관찰은 DNS 수준이다 — IP 직접 접속·DNS-over-HTTPS 회피는 보지 못한다(잡으려면 eBPF 집행 영역으로 가야 하므로 의도적 비범위).
- 플랫폼은 Linux(ubuntu) 러너 먼저 — macOS/Windows 러너는 DNS 후킹 방식이 달라 후속 확장으로 남긴다.
- 네이티브 방화벽이 출시되면 "우리 락파일 → 방화벽 설정 생성기"가 자연스러운 후속이 된다.
- ⓓ(카나리아)는 별도 결정으로 남는다 — 경보 수신 인프라 문제가 풀릴 때 재논의.
